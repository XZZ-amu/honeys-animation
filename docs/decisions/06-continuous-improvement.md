# 阶段 5 设计决策：监控 + 持续改进

## 问题：怎么让系统越跑越好？

不是靠"记住上次的错"就够了，而是要有系统化的数据积累 → 模式发现 → 规则反哺的循环。

## 四个改进方向

| 方向 | 做什么 | 数据来源 |
|------|--------|---------|
| 成功分析 | 做对了什么？提取共性变成可复用的模板/参数 | 用户确认满意的作品 |
| 失败防御 | 不重复犯错 + 诊断原因 + 积累修复路径 | 用户不满意的反馈 + 迭代历史 |
| 参数优化 | 发现"这类动画 + 这组参数 = 高成功率"的模式 | 积累的创作记录 |
| 偏好沉淀 | 提取用户审美偏好，让后续生成更接近用户口味 | 用户的选择行为 + 确认反馈 |

## 改进循环

```
每次创作结束
  ├─ 用户满意 → 分析做对了什么 → 沉淀为模板/偏好
  └─ 用户不满意 → 分析错在哪 → 沉淀为规则/参数调整
                                    ↓
                    积累多了 → 发现模式 → 参数范围收敛
```

## 借鉴 Claude Code 的关键机制

### 1. 全量记录创作过程

Claude Code 做法：Session Transcript 记录每一步（所有 messages、工具调用、结果、metadata），存储在 `~/.claude/sessions/` 下。

我们的对应：每次创作记录完整过程——用户原始描述、意图澄清的选项和选择、生成的 spec、每次生成的结果、用户的反馈、最终确认的版本。这是后续所有分析的原料。

### 2. 结构化事件日志

Claude Code 做法：不是模糊的"成功/失败"，而是结构化事件（哪个工具、什么输入、什么决策、什么结果、耗时多少），精确追踪每个环节。

我们的对应：每次创作标注结构化信息——
- 成功的作品：标注"哪个环节做对了"（意图理解准确？参数合理？配色好？）
- 失败的作品：标注"错在哪个环节"（意图理解偏了？物理参数不真实？改A坏B？）

### 3. 规则即时生效

Claude Code 做法：Hook 让用户"发现问题 → 写一条规则 → 立即生效"，不需要等发版。

我们的对应：`harness/rules/` 目录里的规则文件。发现新的错误模式后，立刻写成规则，下次执行前强制检查。从"发现问题"到"系统约束生效"的周期是分钟级，不是天/周级。

### 4. 区分不同维度的数据

Claude Code 做法：区分工具执行时间 vs 等待用户时间——不区分就会优化错方向。

我们的对应：区分"意图理解的质量"和"执行的质量"。如果系统总是理解对了但做不好 → 优化执行能力。如果总是理解错了 → 优化意图澄清流程。不区分就会优化错方向。

### 5. 策略参数用数据验证

Claude Code 做法：Feature gate + A/B 测试，策略参数不是拍脑袋定的，是用线上数据验证后逐步调优。

我们的对应：比如"选项确认最多问几题"、"熔断阈值是 3 次还是 5 次"、"批量生成出几个版本"——这些参数应该基于积累的数据调整，不是永远固定。

## 存储结构

```
harness/
├── rules/          — 失败教训（防错规则）
├── patterns/       — 成功模式（"这类动画的最佳参数范围"）
├── preferences/    — 用户偏好（审美偏好、风格偏好）
└── logs/           — 创作记录（全量过程日志）
```

## 关键洞察

1. **全量记录是一切改进的前提。** 没有过程数据，就没法事后分析"为什么这次成功/失败"。Claude Code 记录完整 transcript 不是为了 debug，是为了让系统有"记忆"可以学习。

2. **结构化标注比原始记录更有价值。** 一堆原始数据没用，要标注"这次成功是因为意图理解对了"还是"参数碰巧合适"——这决定了经验能不能被复用。

3. **改进的权力应该在用户手中。** Claude Code 的 Hook 让用户自己写规则而不是等开发团队修。我们的 rules/ 目录同理——用户（和系统一起）可以随时添加新规则，不依赖系统更新。

4. **不区分维度就会优化错方向。** 就像 Claude Code 区分"工具耗时"和"用户等待耗时"一样，我们要区分"意图理解质量"和"执行质量"，才能知道该优化哪一层。
