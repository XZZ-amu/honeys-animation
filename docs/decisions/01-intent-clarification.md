# 阶段 1 设计决策：意图澄清机制

## 问题：用户的模糊描述怎么变成精确的执行指令？

裸跑实验暴露了三个问题：
- 用户说"仙女棒"，agent 理解成"烟花爆炸"（意图偏差）
- 颜色、循环等关键参数没有对齐（信息缺失）
- 生成的动画不循环播放（基础约束未声明）

核心矛盾：用户不会（也不应该）写出精确的技术描述，但 agent 需要精确信息才能执行。

## 讨论过的方案

| 方案 | 做法 | Tradeoff |
|------|------|----------|
| A. 结构化模板 | 让用户填表格（发射方式=?、颜色=?、循环=?） | 信息完整，但用户体验像在填工单 |
| B. 精确描述引导 | 教用户写更详细的自然语言描述 | 门槛高，真实用户不会自然地说出精确描述 |
| C. 选项确认 | agent 把模糊输入解读为 2-3 种可能，让用户选 | 用户只需选不需要写，但 agent 生成的选项可能都不对 |

## 最终选择：方案 C（选项确认）

理由：
- 用户操作成本最低（选比写容易）
- 能暴露 agent 的理解偏差（如果选项里没有用户想要的，说明理解差距大）
- 兼容"自己说一个"的逃生口

## 关键洞察

1. **用户选的是情绪，不是参数。** 问颜色时不应该问"粉色还是蓝色"，应该问"温暖浪漫还是神秘魔法"。参数是情绪的翻译产物，不是用户直接操作的对象。

2. **"真实用户怎么能自然说出 B 呢？"** 这句话揭示了一个 harness 设计原则：不要把"精确表达"的负担放在用户身上，系统应该承担"把模糊变精确"的工作。

3. **选项之间的差异应该是情绪/感受层面的，不是技术参数层面的。** 因为用户做创作时，驱动力是"我想传达某种感觉"，不是"我想要 HSL(350, 80%, 60%)"。

## 最终确定的流程

```
用户模糊描述
  → agent 生成 2-3 个选项（区别在于情绪/感受）
  → 用户选一个（或自己说）
  → agent 追问关键参数（也用情绪选项方式）
  → 产出结构化 spec
  → 交给执行 agent
```

## 实现方案

选了方案 C（prompt 模板 + 结构化输出）：
- 设计一个"动画 spec 模板"
- 意图澄清流程的目标是填完这个模板
- 模板作为执行 agent 的输入
