# 阶段 1 设计决策：迭代修改的锁定机制

## 问题：用户修改动画时，怎么保证不破坏已经对的部分？

用户说"加根棒子，粒子快点消失"，agent 可能顺手把配色、循环、背景等已经确认好的东西也改了。

核心矛盾：给 agent 太多自由度会破坏已有成果，太少自由度会让修改不协调（比如锁了轨迹但改了速度，导致粒子飞不到该去的位置）。

## 讨论过的方案

| 方案 | 做法 | Tradeoff |
|------|------|----------|
| A. 全锁 | 除了用户明确要改的字段，其余全部锁定 | 安全但僵硬——联动的细节也被锁了，改完可能不协调 |
| B. 不锁 | 每次修改都把完整 spec 重新传，靠 agent 自觉保持 | 灵活但危险——agent 可能擅自改动用户已确认的东西 |
| C. 分层锁 | 只锁"已被准确传达的意图"，不锁"实现细节" | 平衡点——用户的共识不被破坏，技术参数可以联动调整 |

## 最终选择：方案 C（分层锁）

规则：**锁意图，不锁实现。**

- 🔒 锁定层（用户确认过的意图）：动画类型、情绪/氛围、配色方案、粒子内容、循环方式、背景色
- 🔓 自由层（agent 的实现空间）：具体速度、粒子寿命、轨迹弧度、重力、角度范围、大小变化、数量密度

判断标准：**这个字段的值是用户在选项确认环节明确选择/同意的吗？** 是 → 锁。不是 → 不锁。

## 关键洞察

1. **"锁已经被准确传达的意图，不锁联动的细节"** — 意图是用户和系统之间的共识，不能单方面改。实现细节服务于意图，当意图的某个方面变了，细节理应跟着调整。

2. **锁的依据不是"重要不重要"，而是"有没有被对齐过"。** 一个参数可能很重要（比如粒子数量直接影响视觉密度），但如果用户从没确认过这个值，就不该锁——因为它本来就是 agent 自己定的，没有"对齐"可言。

3. **这跟设计工具的图层锁定不一样。** 图层锁定是"这块区域不能动"（空间锁）。我们的锁是"这个决策不能翻"（语义锁）。

## 执行机制

每次成功生成后，系统保存一份 spec 快照，标注每个字段的来源：
- `source: user_confirmed` → 进入锁定层
- `source: agent_decided` → 进入自由层

用户提修改时：
1. 系统把修改意见解析为"要改哪些字段"
2. 锁定层的字段原样传给执行 agent，标注"已确认，不可修改"
3. 自由层的字段传给执行 agent，标注"可根据修改需要调整"
4. 新的修改如果被用户确认了，也进入锁定层
