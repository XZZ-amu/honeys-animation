# 阶段 3 设计决策：失败修复策略

## 问题：结果不满意时，系统怎么在用户失去耐心前解决问题？

用户对调整的耐心不超过 3-5 次。系统必须在这个窗口内解决问题，而且不能"降低预期"或"让用户换方向"——应该是系统自己变聪明。

## 核心设计：分层修复 + 熔断升级

### 正常路径（第 1-2 次迭代）

用户说"不对"→ 系统从反馈中定位问题类型 → 走对应修复路径：

| 问题类型 | 信号 | 修复路径 |
|---------|------|---------|
| 意图传达遗漏 | "没有棒"、"应该是持续喷射" | 补全 spec 中缺失的要素 |
| 参数不对 | "太慢了"、"存活太长" | 调整自由层参数，锁定其余 |
| 物理不真实 | "不像真的仙女棒" | 查映射表，用真实世界的参数覆盖 |
| 改 A 坏 B | "烟花感觉没了" | 锁住已确认对的部分，缩小修改范围 |

### 熔断升级（第 3 次还不对）

系统不等用户失去耐心，主动做"自我诊断"：

1. **回顾前几次失败**：分析前 2-3 次分别错在哪，找共性
2. **判断卡住的原因**：
   - 每次都是同类问题 → 说明该类问题的修复路径不对，换方式
   - 每次改对 A 但坏 B → 说明锁定机制不够，需要锁更多
   - 反馈越来越模糊 → 说明意图根本没对齐，回阶段 1 重出选项
3. **主动切换策略**（不是问用户"要不要换"）：
   - 意图没对齐 → 出 3 个不同方向的版本让用户选
   - 改 A 坏 B → 把当前最好的版本完整锁住，只开放用户明确要改的一个点
   - 技术方案不行 → 换实现方式（Canvas → CSS、换动画算法）
   - 参数怎么调都不对 → 查看是否有预设的"真实世界参数模板"可以整体替换

## 借鉴 Claude Code 的设计

| Claude Code 的做法 | 我们场景的对应 |
|---|---|
| 每种失败有不同修复路径（不是统一重试） | 意图遗漏/参数不对/改坏了 是三种不同问题，走不同修复路 |
| 失败时注入信息"为什么失败、该怎么做" | 修改时告诉执行 agent"上次错在哪，这次注意什么" |
| 连续 N 次同类失败 → 自动升级策略（熔断器） | 第 3 次不对 → 系统主动分析前几次的共性问题并切换策略 |
| 升级是"换方式"不是"放弃" | 永远不降低预期或让用户换方向，而是系统自己变聪明 |

## 关键洞察

1. **系统要有"自我诊断"能力。** 不是每次失败都当全新问题处理，而是分析"前几次分别错在哪"找共性模式。这跟 Claude Code 的 denialTracking 一样——追踪失败历史，达到阈值后自动升级。

2. **永远不让用户感到"系统放弃了"。** "要不要换方向"、"降低预期"这些话等于承认系统无能。正确做法是系统自己切换策略，让用户感到"它在想办法"。

3. **熔断不是放弃，是升级。** 从"调参数"升级到"换方案"，从"文字迭代"升级到"出 3 个版本选"。每次升级都是在尝试更高维度的解法。

4. **用户的耐心是不可再生资源。** 系统设计的核心约束不是"怎么修对"，而是"怎么在 3-5 次内修对"。超过这个窗口，再好的修复也没意义。
