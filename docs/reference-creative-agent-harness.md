# AI 创作类 Agent：5 个核心难题与成熟解法

面向产品设计师的参考文档。每个阶段回答一个问题：创作类 AI agent 在这一步会遇到什么具体困难，业界有哪些验证过的思路来解决。

---

## 阶段 1：用户脑子里的画面怎么传达给 AI？

### 难题定义

用户脑中有一个具体的视觉想象，但语言描述天然是模糊的。"一只赛博朋克风格的猫在城市里奔跑"——这句话对应几千种可能的画面。难点在于：自然语言的信息带宽远低于视觉信息，光靠文字永远无法精确传达构图、色调、材质、动态。

### 成熟思路

| 思路 | 解决什么 | 怎么用 | 生态中叫什么 | Tradeoff |
|------|----------|--------|--------------|----------|
| **参考图输入** | 风格/构图/角色的精确传达 | 给 AI 一张"我想要像这样"的图，AI 提取其中的特征作为生成约束 | ControlNet（姿态/边缘/深度）、IP-Adapter（风格/角色）、Midjourney 的 image prompt | 依赖你手头有合适的参考图；参考图的"哪些方面"被提取难以精确控制 |
| **提示词工程** | 用结构化语言压缩视觉意图 | 正面描述要什么 + 负面描述不要什么 + 用权重语法强调重点 | Midjourney 的 prompt + --no；SD 的 (keyword:1.3) 权重语法；DALL-E 的自然语言 | 学习成本高、不同模型语法不同；对空间关系和细节的控制力有限 |
| **风格注入** | 把"说不清楚的调性"一次性打包 | 用少量样本图训练一个"风格插件"，之后只需激活插件即可复用风格 | LoRA（轻量微调）、Textual Inversion（学习一个新词代表风格）、Midjourney 的 --sref | 需要 10-30 张风格统一的样本图来训练；风格和内容可能耦合（学了风格也学了不想要的内容） |
| **结构控制** | 精确控制"东西放哪"和"什么区域画什么" | 提供一张草图/分区图，指定每个区域的内容 | Inpainting（局部重画）、Outpainting（向外扩展）、Regional Prompt（分区提示词）、ComfyUI 的 ConditioningSetArea | 控制越精确，创意自由度越低；设置门槛高 |
| **基于结果微调** | 在已有结果上做方向性调整 | 选中一张还行的结果，告诉 AI"保持大体不变，只改 XX" | Midjourney 的 Vary（Subtle/Strong）、SD 的 img2img + 低 denoise | 微调幅度难控制——改少了没变化，改多了面目全非 |
| **多轮对话式细化** | 处理"说不清楚但看到会知道对不对"的情况 | 每轮给 AI 反馈（太冷了、人物再年轻点、背景再简洁些），逐步逼近目标 | ChatGPT + DALL-E 的对话式生成、Midjourney 的 remix mode | 轮次多效率低；AI 可能"忘了"前几轮的约束 |

### 思路之间的关系

这些思路是**分层组合**的，不是互斥的：
- **打底层**：参考图 + 风格注入（定大方向）
- **细化层**：提示词工程 + 结构控制（定细节）
- **迭代层**：微调 + 多轮对话（做最后调整）

最高效的工作流是从粗到细：先用参考图锁定大方向 → 用提示词补充细节 → 用结构控制修正布局 → 用微调做最终调整。跳过前面直接写详细 prompt 是新手常见错误——语言的精确度天花板很低。

### 对应到字符动画场景

字符动画 HTML 的"意图传达"难题体现在：用户想要"小蜜蜂开心地飞向花朵"，但"开心"是什么动作？翅膀振幅大？飞行路径是波浪形？身体有轻微摇晃？

**最适合的思路**：参考图输入（提供动画参考视频/GIF 截帧） + 结构化 spec 模板（角色/动作/时间轴/情绪分开描述）。因为字符动画的变量是有限且可枚举的（路径、速度、节奏、表情、特效），比自由度极高的图片生成更适合用结构化描述取代自然语言。

## 阶段 2：生成结果"好不好"怎么判断？

### 难题定义

创作没有标准答案。同一个 prompt 生成的 10 张图，哪张"最好"取决于主观审美、使用场景、品牌调性等难以量化的因素。难点在于：你需要一个不参与创作的独立"评委"来判断质量，但审美偏好很难编码成规则。

### 成熟思路

| 思路 | 解决什么 | 怎么用 | 生态中叫什么 | Tradeoff |
|------|----------|--------|--------------|----------|
| **美学评分模型** | 自动过滤明显不好看的结果 | 用一个专门训练过的模型给生成结果打分（1-10），低于阈值的自动丢弃 | LAION Aesthetic Predictor、ImageReward | 只能判断"大众审美下好不好看"，无法判断是否符合你的特定意图 |
| **文图匹配度评分** | 判断"生成的东西是不是 prompt 说的东西" | 用 CLIP 模型同时理解图片和文字，计算两者的语义距离 | CLIP Score、Midjourney 的内部排序机制 | 匹配度高不代表好看；模型可能被"欺骗"（技术上匹配但视觉上不对） |
| **批量生成 + 人工选择** | 处理"好不好看我说了算"的主观判断 | 一次生成多个候选（4-16 张），由人快速扫一眼选出最好的 | Midjourney 的 4 宫格、SD 的 batch size | 人工成本高；单次能出的候选数量受算力限制 |
| **硬性规则检查** | 确保满足不可商量的技术要求 | 自动验证分辨率、宽高比、色彩空间、文件大小等硬性指标 | ComfyUI 的 Image Size 节点、自定义 validator 脚本 | 只能检查"技术达标"，无法评估创作质量 |
| **一致性检查** | 判断多个输出之间是否"像同一个项目" | 检测角色外观、配色方案、风格调性在不同帧/镜头之间的偏差 | 目前多为自定义流程：用 CLIP 提取特征后计算余弦相似度 | 阈值难设——太严会把合理的视角变化判为不一致，太松会漏掉明显偏差 |
| **A/B 对比测试** | 在两个方案之间做决策 | 把两种风格/参数的结果并排展示，让用户（或评审团队）投票 | Runway 的 compare 功能、内部评审流程 | 只适用于"二选一"场景；需要有评审人参与 |

### 思路之间的关系

分为**自动过滤 + 人工精选**两层：
1. **自动层**（前三者）：硬性规则检查 → 文图匹配度 → 美学评分，逐步淘汰。作用是把 16 个候选筛到 4 个。
2. **人工层**（后三者）：批量展示 → 人选择 → A/B 对比做最终决策。

一致性检查是**跨输出的全局评估**，不属于单张图的好坏判断，而是判断"这些图放在一起是不是一个项目"。

关键认知：自动评估永远无法取代人的最终判断，但它的价值在于**降低人需要判断的数量**——从 100 张里选 1 张是痛苦的，从 4 张里选 1 张是愉快的。

### 对应到字符动画场景

字符动画的"好不好"有两个层面：单帧好看（静态质量） + 动起来流畅（动态质量）。一个角色静态好看但运动时抖动，或者单帧完美但帧间风格不统一，都是"不好"。

**最适合的思路**：硬性规则检查（帧率、尺寸、字符是否在画布内） + 一致性检查（角色跨帧外观一致） + 人工审片（动态效果只有看了才知道）。自动评估重点放在"一致性"上，因为字符动画最常见的问题不是"不好看"而是"不统一"。

## 阶段 3：结果不满意时，问题出在哪？

### 难题定义

生成结果不满意，但原因可能出在很多环节：prompt 没写清楚？模型本身能力不够？参数设错了？素材质量太低？难点在于：创作管线中变量太多，不做控制变量实验就无法定位根因，而每次实验都要花时间和算力。

### 成熟思路

| 思路 | 解决什么 | 怎么用 | 生态中叫什么 | Tradeoff |
|------|----------|--------|--------------|----------|
| **控制变量法** | 区分"哪个变量导致了问题" | 固定 seed + 只改一个参数，对比前后结果差异 | SD 的 seed 固定 + XY Plot（参数网格对比）、ComfyUI 的 batch 节点 | 每次实验花时间；变量多时组合爆炸 |
| **失败模式分类** | 快速匹配"这种问题通常什么原因" | 建立常见问题-原因的映射表（手崩了→模型问题；颜色偏→prompt 问题；构图乱→缺结构控制） | 社区积累的 troubleshooting 指南、CivitAI 的模型评论区经验 | 依赖经验积累；新问题没有现成分类 |
| **分层诊断** | 系统性排查而不是猜 | 按管线顺序逐层检查：prompt 解析是否正确 → 模型输出是否合理 → 后处理是否引入问题 | ComfyUI 的逐节点预览（每个节点都可以接 Preview 看中间结果） | 需要对管线有基本理解；增加调试设置工作量 |
| **升级策略决策树** | 判断"现在该加什么" | 问题属于哪一类决定下一步：描述不够精确→加 ControlNet；风格不对→加/换 LoRA；模型能力不够→换模型；素材差→重新准备素材 | 经验法则，无标准化工具 | 判断"属于哪一类"本身需要经验 |
| **回退对比** | 确认"是这次改动引起的还是原本就有" | 回到上一个满意的状态，确认问题是新引入的还是原本就存在但没注意到 | 所有版本管理工具（Git）、ComfyUI 的 workflow 历史、Midjourney 的 job 历史 | 需要有保留历史的习惯 |

### 思路之间的关系

诊断的正确顺序是：
1. **回退对比**（先确认问题是不是新引入的）
2. **失败模式分类**（根据现象快速缩小范围）
3. **控制变量法**（在缩小的范围内精确定位）
4. **分层诊断**（控制变量不奏效时，逐层排查中间结果）
5. **升级策略决策树**（定位根因后，选择正确的修复路径）

核心原则：**先缩小范围再精确定位**。最常见的错误是跳过前两步直接开始改参数试错——这就是"盲目重试"。

### 对应到字符动画场景

字符动画的问题诊断相对简单，因为变量可枚举：角色定义 → 动作描述 → 时间线 → CSS/JS 实现。问题出现时，先看是"设计层面错了"（角色描述不对）还是"实现层面错了"（代码逻辑 bug）。

**最适合的思路**：分层诊断（先看最终渲染结果 → 再看中间状态 → 定位到具体帧/动作） + 失败模式分类（角色变形=样式计算错误；动作卡顿=关键帧设置问题；时间不对=时间线逻辑错误）。由于字符动画可以逐帧审查，诊断成本比图片生成低很多。

## 阶段 4：一致性怎么跨帧/跨镜头保持？

### 难题定义

AI 生成模型本质上是"每次独立生成"——它没有"记忆"，不知道上一帧长什么样。要让角色在 100 帧动画中始终是"同一个人"，需要在系统层面强制一致性。难点在于：一致性约束越强，创作灵活度越低；完全不约束则每帧都像不同人画的。

### 成熟思路

| 思路 | 解决什么 | 怎么用 | 生态中叫什么 | Tradeoff |
|------|----------|--------|--------------|----------|
| **角色参考表** | 确保角色外观跨帧一致 | 先生成一张角色的多角度参考图，后续每帧生成时都把这张图作为约束输入 | Character Reference Sheet、IP-Adapter FaceID、Midjourney 的 --cref | 角色只能在参考图已覆盖的角度范围内一致；极端动作/表情可能偏离 |
| **风格锁定** | 确保视觉风格跨帧不飘 | 用同一套风格参数（LoRA + sampler + seed 基数）贯穿所有帧的生成 | Style LoRA、统一 checkpoint、Runway 的 style lock | 风格太锁死会让画面单调；不同场景可能需要不同氛围 |
| **帧间插值** | 确保运动连续性 | 只生成关键帧，中间帧用算法插值而非独立生成 | FILM（帧插值模型）、Runway Gen-2 的 interpolation、AnimateDiff | 插值质量取决于关键帧之间的差异——差异太大则插值失真 |
| **视频扩散模型** | 直接生成连续帧序列 | 不再逐帧生成，而是用 video diffusion 模型一次输出一段连续视频 | Runway Gen-3、Pika、Stable Video Diffusion、Kling | 可控性差（很难精确控制某一帧的内容）；时长受限（通常 4-16 秒） |
| **状态追踪器** | 在系统层面记住"当前场景的所有已确定信息" | 维护一个结构化的状态对象：角色位置、表情、场景元素、已建立的设定，每帧生成前作为 context 输入 | 多为自定义实现；类似游戏引擎的 scene graph | 实现复杂度高；状态越详细维护成本越高 |

### 思路之间的关系

分为两个维度：
- **外观一致性**：角色参考表 + 风格锁定（静态约束）
- **运动一致性**：帧间插值 / 视频扩散（动态约束）
- **语义一致性**：状态追踪器（逻辑约束）

视频扩散模型和帧间插值是**互斥**的（前者一次出连续帧，后者在关键帧之间补帧）。但两者都可以跟角色参考表和风格锁定组合使用。

状态追踪器是**元层面**的——它不直接参与生成，而是告诉生成系统"现在应该生成什么"。其他四种是**执行层面**的——它们直接约束生成过程。

### 对应到字符动画场景

字符动画 HTML 的一致性问题比 AI 图片生成简单很多：角色是代码定义的（CSS/SVG），天然是同一个。真正的一致性挑战在于：**动作风格的统一**（同一个角色在不同场景中的运动节奏应该一致）和**视觉风格的统一**（色彩、线条粗细、阴影方式跨镜头统一）。

**最适合的思路**：状态追踪器（维护角色的 design token——颜色、尺寸、动画曲线等，所有帧引用同一份定义） + 风格锁定（用 CSS 变量/design token 确保视觉参数全局统一）。字符动画的优势是"一致性可以被代码保证"，不需要靠 AI 去"记住"角色长什么样。

## 阶段 5：怎么让出好结果的概率越来越高？

### 难题定义

每次创作都从零开始是低效的。好的创作系统应该能把过去的成功和失败转化为未来的优势。难点在于：创作的"好"是高度场景化的——上次好用的参数换个场景可能完全不适用，经验的迁移性有限。

### 成熟思路

| 思路 | 解决什么 | 怎么用 | 生态中叫什么 | Tradeoff |
|------|----------|--------|--------------|----------|
| **Prompt 库** | 避免每次重新发明 prompt | 把验证过的好 prompt 归档、打标签、做可检索，下次类似需求直接调用或微调 | CivitAI 的 prompt 分享、Midjourney 的 /describe 逆向、PromptHero 等 prompt 市场 | 别人的好 prompt 未必适合你的模型/场景；库大了之后搜索和维护成本高 |
| **参数模板** | 不同场景有不同的"最佳配方" | 为常见场景（人像/风景/UI 截图/动画帧）整理一套验证过的参数组合，一键应用 | ComfyUI 的 workflow template、SD WebUI 的 styles 功能、Runway 的 preset | 模板会过时（模型更新后参数含义变化）；容易形成路径依赖不尝试新配置 |
| **失败案例库** | 避免重复踩坑 | 记录"什么 prompt + 什么参数 = 什么问题"，下次遇到类似需求先查是否有已知陷阱 | 无标准化工具，多为个人笔记/团队 wiki | 维护成本高；问题是否"类似"的判断需要经验 |
| **人类反馈积累** | 让选择越来越准 | 每次人工选择"好/不好"时记录偏好，用这些偏好数据训练排序模型或调整生成参数 | RLHF / DPO 的简化版、Midjourney 的用户投票系统（影响模型迭代） | 需要大量标注数据才能见效；偏好会随时间变化 |
| **风格资产库** | 验证过的"风格插件"可复用 | 把训练好的 LoRA、验证过的 checkpoint 组合、调好的后处理链作为资产管理 | CivitAI 的模型库、HuggingFace 的 model hub、团队内部的 asset library | 资产间可能冲突（两个 LoRA 同时用效果打架）；需要版本管理 |

### 思路之间的关系

这五种思路构成一个**知识管理系统**的不同层面：
- **输入端**：Prompt 库 + 参数模板（"用什么生成"的经验）
- **输出端**：人类反馈积累（"什么算好"的标准）
- **工具端**：风格资产库（"用什么工具"的选择）
- **防守端**：失败案例库（"什么不能做"的边界）

最高效的组合是：参数模板（快速启动） + 失败案例库（避免已知坑） + 人类反馈（持续校准方向）。Prompt 库和风格资产库是锦上添花——有了前三者，即使没有精心维护的库也能逐步提升。

关键认知：**持续改进的核心不是积累更多资产，而是建立反馈闭环**——每次创作的结果（好或不好）都要被系统捕获、分类、用于下次决策。没有闭环，再多的资产也只是堆积不是改进。

### 对应到字符动画场景

字符动画 HTML 的持续改进优势在于：代码是可版本化的。每次成功的动画效果可以直接作为组件复用，不像 AI 图片生成那样"同样的 prompt 下次结果不同"。

**最适合的思路**：参数模板（不同动作类型的标准 timing 和 easing 曲线） + 失败案例库（哪些 CSS 动画属性组合会导致性能问题/渲染异常） + 组件化复用（验证过的动画效果封装为可复用组件）。字符动画可以做到真正的"确定性复用"——同样的代码永远产出同样的结果，这是 AI 生成做不到的。

## 总结：5 个阶段的核心认知

| 阶段 | 核心难题 | 关键认知 |
|------|----------|----------|
| 1 意图传达 | 语言的信息带宽低于视觉 | 从粗到细分层控制，不要试图一句 prompt 搞定一切 |
| 2 质量评估 | 审美无法完全自动化 | 自动评估的价值是减少人需要判断的数量，不是取代人 |
