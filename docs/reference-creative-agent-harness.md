# ComfyUI Harness 拆解：创作类 Agent 的 5 阶段参照

以 ComfyUI（Comfy-Org/ComfyUI）的实际源码为参照物，拆解一个成熟创作类 agent harness 的设计。每个阶段指出具体机制、设计原因、缺失后果，并与 Claude Code 对比。

---

## 阶段 1：任务理解 + 工具就绪

**让系统知道做什么、有什么可用。**

### 具体机制

ComfyUI 用三层结构让系统理解"用户要什么"和"能用什么工具"：

1. **节点图即任务描述**（`DynamicPrompt` 类，`comfy_execution/graph.py`）— 用户在前端拖拽节点、连线，形成一个有向无环图（DAG）。这个图被序列化为 JSON 发送给后端。`DynamicPrompt` 类接收这份 JSON，成为整个执行引擎理解任务的基础。比如用户拖了一个 "Load Checkpoint" 节点连到 "KSampler" 再连到 "Save Image"，这三个节点加上连线就是完整的"任务描述"。

2. **工具注册表**（`NODE_CLASS_MAPPINGS`，`nodes.py`）— 全局字典，记录所有可用节点的类名到 Python 类的映射。每个节点类通过四个属性自我描述能力：`INPUT_TYPES()`（需要什么输入）、`RETURN_TYPES`（输出什么）、`FUNCTION`（执行哪个方法）、`CATEGORY`（属于哪个分类）。系统启动时扫描 `comfy_extras/` 和 `comfy_api_nodes/` 目录，自动注册所有节点。

3. **预执行验证**（`validate_prompt()` 函数，`execution.py` 第 1106 行）— 在任何节点开始执行之前，系统做完整的合法性检查：
   - 每个节点是否存在于注册表中（找不到就报 `missing_node_type`）
   - 图中是否有输出节点（没输出就拒绝执行）
   - 每个连线的类型是否匹配（`validate_node_input()` 做类型兼容检查）
   - 是否有循环依赖（`visiting` 列表检测图中的环）
   - 必填输入是否都有值（`required_input_missing` 错误）

4. **动态扩展机制**（`ephemeral_prompt`，`DynamicPrompt` 类）— 图不是静态的。执行过程中，节点可以通过返回 `expand` 来动态创建新节点，系统用 `add_ephemeral_node()` 把它们加入图中。这让"工具集"在运行时也能扩展。

### 为什么这么设计

创作类任务不适合用自然语言描述。"帮我生成一张赛博朋克风格的猫"看起来简单，但背后涉及模型选择、采样参数、后处理步骤等十几个决策。节点图把这些决策全部显式化、结构化。验证前置则保证了"不合法的任务连开始都不会开始"——发现问题的成本远低于执行一半再失败。

### 如果没有这层

用户连了一个错误的线（比如把图像输出接到文本输入），系统会等到执行到那个节点才报错。之前已经花了 30 秒加载模型、跑了一半的采样，全部白费。更糟的是错误信息含糊——用户不知道是哪个连接出了问题。

### 跟 Claude Code 的对比

| 维度 | Claude Code | ComfyUI |
|------|-------------|---------|
| 任务描述形式 | 自然语言 + CLAUDE.md 约束 | 结构化节点图（JSON DAG） |
| 工具注册 | Deferred Tools + ToolSearch（按需加载） | NODE_CLASS_MAPPINGS（启动时全注册） |
| 验证时机 | 执行中通过 hook 拦截 | 执行前一次性全验证 |
| 能力边界 | permissions allow/deny/ask | 类型系统（输入输出类型必须匹配） |

**为什么不同**：Claude Code 面对开放式任务，无法提前验证"用户的要求是否合法"。ComfyUI 的任务是封闭的——节点图里所有信息都是确定的，所以可以在执行前做完整验证。这是"开放式对话 agent"和"结构化工作流引擎"的根本差异。

## 阶段 2：执行 + 分离评估

**做事的和判断"要不要做"的必须是不同机制。**

### 具体机制

1. **拓扑排序执行**（`ExecutionList` 类，`comfy_execution/graph.py`）— 系统不是随便挑节点执行，而是按依赖关系排序：所有前置节点完成后，当前节点才变为"就绪"。`stage_node_execution()` 方法从就绪列表中挑选下一个执行的节点，`complete_node_execution()` 在完成后释放被它阻塞的下游节点。

2. **变更检测与缓存判定分离**（`IsChangedCache` 类，`execution.py`）— 执行前，系统先问每个节点："你的输入变了吗？"节点通过 `IS_CHANGED` 方法回答。如果没变，`CacheKeySetInputSignature` 会计算出相同的缓存签名，系统直接跳过该节点。这个"判断是否需要重新执行"的逻辑独立于"实际执行"逻辑。

3. **UX 优先的执行顺序**（`ux_friendly_pick_node()` 方法）— 在多个节点都就绪时，系统优先挑选输出节点（比如 PreviewImage），让用户尽早看到结果。这是"评估什么对用户最重要"的逻辑，与"按正确顺序执行"分离。

4. **Lazy Evaluation 机制**（`check_lazy_status` 方法）— 节点可以声明某些输入是"懒加载"的。执行到该节点时，系统先调用 `check_lazy_status` 让节点自己决定"我真的需要这个输入吗？"如果不需要，上游节点就不用执行。这是节点自身对"是否执行上游"的评估权。

5. **ExecutionBlocker 阻断机制**（`ExecutionBlocker` 类，`comfy_execution/graph_utils.py`）— 当某个节点返回 `ExecutionBlocker` 时，它的所有下游节点都收到阻断信号而不是正常输出。系统不会因为一个节点失败就终止整个图的执行——无关分支仍然可以继续。

### 为什么这么设计

创作类管线的执行时间长（生成一张图可能要 30 秒到几分钟）。如果每次都从头执行所有节点，调参体验会极差。把"判断是否需要执行"和"实际执行"分开，让系统只重做真正变化了的部分。`IsChangedCache` 就是那个独立的"评估者"——它不执行任何生成逻辑，只负责判断"这个节点的输入签名跟上次一样吗"。

### 如果没有这层

用户改了一个采样参数（比如 steps 从 20 改到 30），系统会把模型重新加载一遍、CLIP 编码重新跑一遍——尽管这些跟采样参数无关。在复杂工作流中（20+ 节点），这意味着每次微调都要等几分钟，调参变成煎熬。

### 跟 Claude Code 的对比

| 维度 | Claude Code | ComfyUI |
|------|-------------|---------|
| 评估者角色 | 子 Agent + edit-gate hook | IsChangedCache + CacheKeySetInputSignature |
| 评估什么 | "这个修改方案对不对" | "这个节点需不需要重新执行" |
| 阻断机制 | hook 拦截编辑动作 | ExecutionBlocker 阻断下游传播 |
| 粒度 | 整个任务级（做还是不做） | 节点级（每个节点独立判断） |

**为什么不同**：Claude Code 的"分离评估"是防止 agent 犯错（执行者容易高估自己的正确性）。ComfyUI 的"分离评估"是防止重复计算（执行者不需要知道自己是否该被跳过，由缓存系统统一判断）。前者是质量控制，后者是效率优化——但底层逻辑一样：做事的不该自己判断该不该做。

## 阶段 3：失败修复策略

**不是简单重试，而是精确定位问题并隔离影响范围。**

### 具体机制

1. **精确定位错误节点**（`handle_execution_error()` 方法，`execution.py`）— 当节点执行失败时，系统生成结构化错误信息，包含：`node_id`（哪个节点）、`exception_type`（什么类型的错误）、`current_inputs`（当时的输入值）、`traceback`（调用栈）。前端可以精确高亮出错的节点，用户知道该去修哪里。

2. **错误不扩散，分支隔离**（`ExecutionBlocker` + 执行循环的 `break` 逻辑）— 当一个节点失败时，系统记录错误后跳出执行循环，但已经成功执行的节点结果被缓存保留。下次用户修复问题后重新运行，成功的部分直接用缓存，只重跑失败节点及其下游。

3. **分类错误提供针对性建议**（`execution.py` 第 630 行附近）— 系统识别特定错误模式并给出修复建议。比如 OOM（显存不足）时提示"检查 batch_size 是否设太大了"；矩阵维度不匹配时提示"检查 CLIP Loader 的文件和类型选择"。这不是盲目重试，而是根据错误类型指向根因。

4. **依赖循环专项处理**（`DependencyCycleError`，`get_nodes_in_cycle()` 方法）— 如果执行过程中出现循环依赖（通常由动态扩展产生），系统会定位循环中的责任节点（优先 blame 动态创建的节点而非用户手动放置的节点），而不是笼统地说"图有问题"。

5. **类型不匹配的预防性拦截**（`validate_node_input()` 函数，`comfy_execution/validation.py`）— 通过 union type 系统（`STRING,INT` 这样的复合类型），在连线时就判断兼容性。支持 strict 模式（接收类型必须是输入类型的子集）和宽松模式（有交集即可），平衡了严格性和灵活性。

### 为什么这么设计

创作类管线的失败粒度是"节点级"而非"任务级"。一个 20 节点的工作流，可能只有第 15 个节点（比如 ControlNet 的 preprocessor）出了问题。如果系统只能说"执行失败了"，用户要从头排查 20 个节点。精确到节点+输入+建议的错误报告，让修复变成"看到红色节点 → 看错误提示 → 改那个参数"的简单流程。

### 如果没有这层

用户会反复尝试"改这试试、改那试试"——跟 Claude Code 里 agent 的试错式修复一样的问题，只不过是人在试错。更糟的是，如果没有分支隔离，一个节点失败后所有缓存失效，修复后要从头跑整个流程。对于需要 5 分钟生成一张图的复杂工作流，这是不可接受的。

### 跟 Claude Code 的对比

| 维度 | Claude Code | ComfyUI |
|------|-------------|---------|
| 修复主体 | agent 自己（受 diagnose skill 约束） | 用户（系统提供定位信息） |
| 错误粒度 | 整个操作成功/失败 | 精确到节点 + 输入 + 错误类型 |
| 隔离策略 | 不适用（单线程执行） | 失败节点下游阻断，无关分支不受影响 |
| 防重复失败 | 铁律"禁止试错式修复" | 缓存保留成功部分，只重跑失败链路 |

**为什么不同**：Claude Code 是让 agent 自修复，所以需要强制诊断流程防止盲改。ComfyUI 是让人修复，所以重点是"给人足够的信息做出正确判断"。两者共同的设计哲学是：修复前必须先定位根因，不能靠反复尝试。

## 阶段 4：Context 生命周期管理

**长时间运行中保护中间产物、管理资源消耗。**

### 具体机制

1. **三级缓存策略**（`CacheSet` 类，`execution.py`）— 系统提供三种缓存模式：
   - **Classic**（默认）：用 `HierarchicalCache` + `CacheKeySetInputSignature`，基于输入签名缓存。同一个节点只要输入没变，输出就直接复用。prompt 结束后释放。
   - **LRU**：最近使用的 N 个结果保留，超出的按使用时间淘汰。适合反复在几个参数间对比的场景。
   - **RAM Pressure**：监控系统内存压力（`psutil.virtual_memory().available`），只在内存紧张时才释放缓存。最大化利用可用资源。

2. **输入签名 = 缓存键**（`CacheKeySetInputSignature` 类，`comfy_execution/caching.py`）— 缓存的 key 不是"节点 ID"，而是"这个节点的所有祖先节点的类型+输入值的完整签名"。这意味着：即使用户删掉一个节点再重新创建（ID 变了），只要连接和参数一样，缓存依然命中。

3. **Hierarchical 子缓存**（`ensure_subcache_for()` 方法）— 动态扩展产生的临时节点（ephemeral nodes）有自己的子缓存空间，不会污染主缓存。当扩展图被替换时，子缓存随之清理。

4. **RAM 压力释放回调**（`ram_release_callback`，`execution.py` 第 700 行附近）— 在每个节点执行完后检查内存。如果超出阈值，先释放不活跃的缓存（`ram_inactive_headroom`）；如果还不够，释放活跃缓存并调用 `free_pins()` 回收 GPU 内存。

5. **执行历史持久化**（`PromptQueue.history`）— 每次执行完成后，结果（包括状态、输出、时间戳）被存入 history 字典。即使缓存被清理了，用户仍然可以回溯历史执行的结果。

### 为什么这么设计

创作类管线的中间产物很大（一张图几十 MB、一个模型几 GB），不可能全部保留。但频繁的重新计算又很昂贵（加载模型可能要 10 秒）。三级缓存让用户根据自己的硬件条件选择平衡点：内存充足就用 RAM Pressure（尽量多缓存），内存紧张就用 Classic（执行完就释放）。输入签名作为缓存键则确保了"内容相同就复用"的语义正确性。

### 如果没有这层

两种极端后果。如果不缓存：用户改了最后一个节点的参数，整个图从头执行，等 5 分钟看一张图。如果只缓存不释放：几个复杂工作流跑下来，16GB 内存全满，系统开始 swap 到硬盘，速度从分钟级变成小时级，最终 OOM crash。

### 跟 Claude Code 的对比

| 维度 | Claude Code | ComfyUI |
|------|-------------|---------|
| 管理对象 | 对话上下文（token） | 计算结果（张量、图像） |
| 压力指标 | context window 使用率 | RAM 可用量 |
| 释放策略 | compact（压缩摘要） | 三级策略（Classic/LRU/RAM Pressure） |
| 跨 session | memory 系统（结构化记忆） | execution history（结果回溯） |
| 缓存键 | 不适用 | 输入签名（内容寻址） |

**为什么不同**：Claude Code 管理的是"语义信息"——压缩时丢失的是细节和上下文。ComfyUI 管理的是"计算结果"——释放时丢失的是执行时间（需要重新计算）。所以 Claude Code 的核心矛盾是"精度 vs 容量"，ComfyUI 的核心矛盾是"速度 vs 内存"。这决定了前者需要"智能摘要"，后者需要"智能淘汰"。

## 阶段 5：监控 + 持续改进

**追踪执行状态，让用户和系统都能从历史中学习。**

### 具体机制

1. **实时进度推送**（`ProgressRegistry` 类，`comfy_execution/progress.py`）— 每个节点有四种状态：Pending → Running → Finished/Error。执行引擎在关键时刻调用 `start_progress()` / `update_progress()` / `finish_progress()`，通过 WebSocket 实时推送给前端。用户能看到"当前在执行哪个节点、进度多少"。

2. **多 Handler 架构**（`ProgressHandler` 抽象类）— 进度信息同时发送给多个消费者：`WebUIProgressHandler`（推给浏览器前端显示进度条）和 `CLIProgressHandler`（在命令行显示 tqdm 进度条）。新的监控消费者只需实现 `start_handler` / `update_handler` / `finish_handler` 三个方法即可接入。

3. **Job 状态追踪**（`JobStatus` 类，`comfy_execution/jobs.py`）— 每个提交的 prompt 是一个 Job，有完整的生命周期：`PENDING` → `IN_PROGRESS` → `COMPLETED` / `FAILED` / `CANCELLED`。带时间戳（`execution_start_time`、`execution_end_time`），可以回答"这个工作流跑了多久"。

4. **执行历史 + 输出预览**（`normalize_history_item()`，`normalize_outputs()`）— 历史记录包含每个 Job 的输入 prompt、输出结果、执行状态。`preview_output` 机制提取可预览的媒体文件（图片、视频、音频、3D），让用户快速回顾"上次生成了什么"。

5. **缓存命中通知**（`execution_cached` 事件）— 执行开始时，系统广播"这些节点因为缓存命中而被跳过了"。这让用户理解"为什么这次比上次快"——不是出了 bug 跳过了步骤，而是聪明地复用了之前的结果。

### 为什么这么设计

创作类任务执行时间长，用户需要知道"现在在干什么、还要等多久"。没有进度反馈，用户无法区分"正在计算"和"系统挂了"。Job 状态和历史记录则支持了工作流的迭代改进——用户可以对比不同参数的生成结果，找到最优配置。

### 如果没有这层

用户提交一个工作流后，屏幕上什么都不动，不知道还要等 10 秒还是 10 分钟。失败时也没有历史可查——"上次那组效果不错的参数是什么来着？"无从回答。缓存命中时用户以为"怎么这么快，是不是出错了"，对系统产生不信任。

### 跟 Claude Code 的对比

| 维度 | Claude Code | ComfyUI |
|------|-------------|---------|
| 监控对象 | agent 的行为模式（犯了什么错） | 执行状态（跑到哪了、花了多久） |
| 改进方式 | feedback memory → 下次约束行为 | 执行历史 → 用户对比调参 |
| 触发时机 | 对话结束时复盘 | 实时推送 + Job 完成时归档 |
| 改进主体 | harness 设计者（修改 hook/rule） | 用户（调整工作流参数） |

**为什么不同**：Claude Code 的"持续改进"是改进 agent 本身的行为——通过记录错误模式来防止重犯。ComfyUI 的"持续改进"是改进用户的工作流——通过提供执行数据帮用户做更好的创作决策。前者是"系统学习"，后者是"辅助人学习"。这反映了两者定位的差异：Claude Code 追求 agent 自主性越来越强；ComfyUI 追求人的创作效率越来越高。

## 对我们动画项目的启示

### 阶段 1：任务理解 — 动画也需要结构化 Spec

**借鉴**：动画生成不应该用一句话描述（"帮我做一个小蜜蜂飞舞的动画"），而应该有结构化的 spec：角色定义 → 场景设置 → 动作时间轴 → 风格参数。就像 ComfyUI 的节点图，每个决策都显式化。

**具体做法**：设计一个"动画 Spec Schema"，包含：characters（角色列表+描述）、scenes（场景序列）、timeline（动作时间轴）、style（视觉风格参数）。Harness 在执行前验证 spec 的完整性（角色有没有出现在场景里？时间轴有没有冲突？）。

### 阶段 2：执行 — 增量执行是必需品

**借鉴**：ComfyUI 的 `IsChangedCache` 思路直接适用于动画。角色设计没变 → 不重新生成角色图；台词没变 → 不重新生成口型；只有改了的帧需要重做。

**具体做法**：为每个生成步骤建立输入签名。角色签名 = hash(角色描述 + 风格参数)；帧签名 = hash(角色签名 + 动作描述 + 场景背景)。签名不变则复用缓存结果。

### 阶段 3：失败修复 — 帧级别的错误隔离

**借鉴**：ComfyUI 的 `ExecutionBlocker` 思路。第 5 帧生成失败不应该影响前 4 帧和后面的帧（如果它们不依赖第 5 帧的结果）。错误信息要精确到"哪一帧、什么原因"。

**具体做法**：动画管线的错误处理粒度应该是帧/镜头级。失败帧标记为 blocked，用户可以单独重试该帧，或调整该帧的参数后重新生成，而不是整个动画从头来。

### 阶段 4：Context 管理 — 角色一致性是最宝贵的"缓存"

**借鉴**：ComfyUI 的 RAM Pressure 策略——优先保留重新计算成本高的缓存。在动画场景中，"角色一致性"（让同一个角色在所有帧中看起来是同一个人）是最昂贵的计算，应该最后才释放。

**具体做法**：缓存优先级排序：角色 reference sheet > 场景背景 > 关键帧 > 中间帧。内存紧张时先释放中间帧（重新生成最快），角色 reference 尽可能常驻。

### 阶段 5：监控 — 实时预览 + 结果对比

**借鉴**：ComfyUI 的 `WebUIProgressHandler` 实时推送中间结果。动画生成中用户应该能看到"当前正在生成第几帧"和每帧的预览图。

**具体做法**：
- 每帧生成完立即推送预览到前端，用户可以在生成过程中就发现问题并中断
- 保留历史执行记录，支持"这组风格参数 vs 那组风格参数"的并排对比
- 角色一致性检查的结果实时反馈（"第 8 帧的角色跟 reference 偏差较大"）
