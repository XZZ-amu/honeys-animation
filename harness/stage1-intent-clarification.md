# 阶段 1：意图澄清流程

当收到用户的动画创作需求时，系统必须按以下流程执行。不允许跳过。

---

## 流程

```
用户输入（文字/草图/参考图）
  │
  ├─ 判断输入类型
  │   ├─ 纯文字 → 进入选项确认
  │   ├─ 草图 → 提取结构意图，不提取风格
  │   └─ 参考图 → 判断用户想参考什么（见下方）
  │
  ├─ 选项确认（最多 1-2 题）
  │   ├─ 第一题必须区分"结构/形态"（长什么样），不是"运动方式"
  │   ├─ 用户选一个，或说"都不是"
  │   └─ 如果都不是 → 请用户补充描述或给草图
  │
  ├─ 情绪确认（1 题）
  │   ├─ 选项之间的区别是情绪/感受，不是技术参数
  │   └─ 用户选一个，或自己说一个感觉
  │
  └─ 产出填好的 spec（见 animation-spec-template.md）
      → 交给执行 agent
```

## 参考图处理规则

用户给参考图时，根据附带的话判断意图：
- "做一个跟这个一样的，但改 XX" → 复刻结构 + 换指定元素
- "我喜欢这个配色" → 只提取色板
- "参考它这个 XX 效果" → 只提取指定的局部特征
- 什么都没说 → 反问"你想参考它的什么？"

**从参考图提取的原则**：
- 视觉风格（配色、造型语言）→ 复刻
- 物理行为（飞多远、多快）→ 不照抄图片布局，遵循现实世界

## 草图处理规则

从草图中只提取：
- 结构（什么在什么上面、怎么连接）
- 布局（大概的位置和比例关系）
- 元素类型（什么形状、多少个）

不从草图提取：
- 线条风格（草图的粗糙线条不是用户要的最终风格）
- 精确尺寸（草图比例是随手画的，不是精确的）
- 颜色（除非草图明确上了色）

## 选择题设计规则

- 总数不超过 2 题
- 选项 2-3 个 + "自己说一个"逃生口
- 第一题区分结构形态
- 第二题区分情绪/感受
- 不用选择题收集参数（参数由系统推导）

## 参数推导规则

用户不操心参数，系统自动推导：
- 物理参数（速度、寿命、幅度、重力）→ 从"这是什么物体"的现实行为推导
- 视觉参数（颜色、发光、材质）→ 从"用户选的情绪"推导
- 两者独立，不互相污染

---

## 修改阶段的意图确认

意图澄清不只发生在第一次创作，修改细节时同样需要。根据修改的性质分三级处理：

### 直接执行（不打扰用户）

触发条件：纯参数调整，没有歧义。
- "速度快一点"
- "颜色换成蓝色"
- "字母再大一些"

### 出选项确认（效果有歧义）

触发条件：修改涉及"效果/方式"的变化，同一句话能理解出多种做法。
- "换成沙粒消散的感觉" → 问用户是哪种沙粒消散
- "加点弹性" → 弹跳？拉伸回弹？果冻晃动？
- "让它更自然" → 自然在哪个维度？节奏？轨迹？时间差？

做法：给 2-3 个选项 + 简短描述差异，用户选一个。

### 告知并确认（重大实现变更）

触发条件：为了满足修改需求，需要改动底层架构/渲染方式，可能影响已经满意的部分。
- "为了做沙粒碎裂效果，字母的渲染方式要从直接画字变成像素点集合。已有的风速和节奏不会变，但文字的视觉质感会有细微差异。可以吗？"
- "为了加这个效果，需要把物理引擎从简单阻尼换成 Verlet 积分。动画的整体感觉可能会变化。要试试吗？"

做法：先说"我打算怎么做"+"会影响什么"+"不会影响什么"，用户说 OK 再动手。

### 判断流程

```
用户提修改意见
  │
  ├─ 是纯参数调整？ → 直接执行
  │
  ├─ 涉及效果/方式变化？
  │   └─ 这个效果有多种理解方式？ → 出选项确认
  │
  └─ 需要底层架构变更？
      └─ 可能影响已满意的部分？ → 告知并确认
```
