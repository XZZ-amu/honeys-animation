# 阶段 3：失败修复策略

用户说"不对"时，系统的诊断和修复流程。核心约束：3-5 次内必须解决。

---

## 正常路径（第 1-2 次迭代）

用户反馈 → 系统分类问题类型 → 走对应修复路径：

| 问题类型 | 信号示例 | 修复路径 |
|---------|---------|---------|
| 意图传达遗漏 | "没有棒"、"应该有个伞盖" | 补全 spec 中缺失的要素 |
| 参数不对 | "太快了"、"太大了"、"存活太长" | 调整自由层参数，锁定层不动 |
| 物理不真实 | "不像真的风铃"、"风太大了" | 查 rules/01，用真实世界参数覆盖 |
| 改 A 坏 B | "烟花感觉没了"、"之前的 XX 不见了" | 识别被破坏的部分，锁住它，缩小修改范围 |
| 风格不一致 | "跟之前那个不像一个系列" | 检查 style-tokens.css 是否被遵守 |

## 熔断升级（第 3 次还不对）

系统不等用户失去耐心，主动做自我诊断：

### 步骤

1. **回顾前 2-3 次失败**：分别错在哪？
2. **找共性**：
   - 每次都是同类问题 → 这个修复路径本身有问题，换方式
   - 每次改对 A 但坏 B → 锁定机制不够，需要锁更多
   - 反馈越来越模糊 → 意图根本没对齐
3. **主动切换策略**（不是问用户"要不要换"）：

| 诊断结论 | 升级策略 |
|---------|---------|
| 意图没对齐 | 出 3 个不同方向的版本让用户选 |
| 改 A 坏 B | 把当前最好的版本完整锁住，只开放用户明确要改的一个点 |
| 技术方案不行 | 换实现方式（Canvas → CSS、换动画算法） |
| 参数怎么调都不对 | 查是否有"真实世界参数模板"可整体替换 |
| 物理引擎不匹配 | 布局优先重新适配（参考 rules/04） |

## 绝对禁止

- 让用户"降低预期"
- 建议用户"换个方向"
- 无脑重试同样的策略
- 超过 5 次还没解决却继续同样的路径

## 迭代锁定规则

每次修改时执行：
- 🔒 锁定层：用户在选项确认环节明确选择/同意过的字段（不可修改）
- 🔓 自由层：agent 为实现意图自己决定的技术参数（可随修改调整）

判断标准：这个字段的值是用户对齐过的吗？是 → 锁。不是 → 不锁。
